## Root cause: airflow_dag_extract

Upstream `littrs-ruff-python-parser` 0.6.2 still panics in
`expression.rs:1633:25` (`unreachable!()` on cross-typed `FStringMiddle`
/ `TStringMiddle` token routing). This is the **same upstream bug**
already documented in `fuzz/known-crash/airflow_dag_extract/FINDING.md`
from 2026-05-03.

Production callers (`extract_all_static_dags`) route through
`crate::panic_safe::parse_module_safely`, which wraps
`ruff_parser::parse_module` in `std::panic::catch_unwind`. A direct
standalone reproducer confirms production behaviour:

```
err: Internal("ruff_python_parser panicked on input: internal error:
  entered unreachable code: f-string: unexpected token TStringMiddle …")
```

i.e. **production is safe** — the panic is converted to
`ParseError::Internal`.

### Why fuzz still aborts

`libfuzzer-sys` 0.4.x (current 0.4.12) installs a global panic hook in
`initialize()` that calls `process::abort()` on every panic, **before**
any `catch_unwind` boundary further up the stack runs (see
`/home/y1/.cargo/registry/.../libfuzzer-sys-0.4.12/src/lib.rs:81-97`).

That overrides the production `panic_safe` shim's `catch_unwind` —
the shim **does** propagate the panic as `Err(Internal)`, but the
host process is dead first. Result: the fuzz target crashes on inputs
production handles cleanly.

## Fix

`fuzz/fuzz_targets/airflow_dag_extract.rs` now installs a no-op panic
hook (`std::panic::set_hook(Box::new(|_info| {}))`) once at startup via
`std::sync::Once`. This restores parity between fuzz and production:
the production `catch_unwind` runs to completion and returns the
expected `Err(Internal)`. A genuine escape (the shim being bypassed by
e.g. SIGSEGV from stack overflow on a path the bracket-depth cap
misses) still aborts via the non-panic code path, which is what we
actually want to detect.

- `fuzz/fuzz_targets/airflow_dag_extract.rs` — +33 / -3 LOC; pure fuzz
  harness change, **no production code touched**.
- 60s smoke (393,628 runs) on the patched target: 0 crashes.
- Both crash artifacts re-execute in 1 ms, error returned cleanly.
